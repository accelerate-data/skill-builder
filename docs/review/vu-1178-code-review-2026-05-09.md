## Intent

Refactor Eval Workbench into an authoring-only surface where:
1. Scenarios and assertions are SQLite-backed (not YAML/disk-backed).
2. Generation uses the selected-skill OpenHands conversation via `send_openhands_message`.
3. All app-owned eval-run execution concepts are removed (no run status, history, cancellation, or Promptfoo sidecar).

## Findings

### High

1. **[Skeptic] `workspace_run_dir` set to `workspace_root_dir` in generation config**
   - **File:** `app/src-tauri/src/commands/eval_workbench/mod.rs:415-416`
   - **Issue:** The `OpenHandsRuntimeConfigParams` for generation sets both `workspace_root_dir` and `workspace_run_dir` to `runtime_ctx.workspace_path.clone()`. In the old throwaway flow, `workspace_run_dir` was a unique temp directory under `.openhands/throwaway/`. Now it points to the workspace root. This maps to `workspace_skill_dir` in `SidecarConfig` (line 195 of `sidecar.rs`). If the OpenHands agent uses this path for file operations during generation (it has `file_editor` and `terminal` tools allowed), it could write files directly to the workspace root.
   - **Risk:** The agent might create/modify files in the workspace root during scenario generation, which is unexpected for a "send message on existing conversation" flow.
   - **Recommendation:** Verify that `SendExistingOnly` dispatch does not use `workspace_skill_dir` for file operations, or pass a safe isolated path. If the existing conversation already has its own workspace context, this field may be harmless — but it should be documented or set to a neutral value.

2. **[Skeptic] Listener leak if `send_openhands_message` fails before `wait_for_turn_result`**
   - **File:** `app/src-tauri/src/commands/eval_workbench/mod.rs:432-434`
   - **Issue:** `setup_turn_listeners` is called before `send_openhands_message`. If `send_openhands_message` returns an `Err`, the function propagates the error immediately without calling `app.unlisten()` on the registered listeners. The `TurnListener` struct is created but never consumed by `wait_for_turn_result`, so its listeners remain registered for the lifetime of the app.
   - **Risk:** Repeated generation failures would accumulate orphaned event listeners, potentially causing memory growth or spurious event handling.
   - **Recommendation:** Wrap the send+wait in a guard that cleans up listeners on any error path, or use a `Drop` impl on `TurnListener` that calls `unlisten`.

### Medium

3. **[Architect] No UNIQUE constraint on `(plugin_slug, skill_name, name)` in scenarios table**
   - **File:** `app/src-tauri/src/db/migrations.rs:346-356`
   - **Issue:** The `scenarios` table has `id TEXT PRIMARY KEY` but no unique constraint on the `(plugin_slug, skill_name, name)` triplet. The `delete_scenario` function deletes by name lookup, and `read_scenario` also queries by name. If two scenarios with the same name exist (possible via direct DB manipulation or a race), delete/read operations would affect arbitrary rows.
   - **Recommendation:** Add `UNIQUE(plugin_slug, skill_name, name)` to the migration, or change the command layer to use `id` for delete/load operations.

4. **[Minimalist] `save_scenario` always resets `sort_order` to 0**
   - **File:** `app/src-tauri/src/db/eval_workbench.rs:80`
   - **Issue:** The `save_scenario` function hardcodes `sort_order` to `0i64` on every INSERT/UPDATE. The list query sorts by `sort_order ASC, name ASC`, so all scenarios effectively sort by name only. If sort ordering is intended to be user-controllable in the future, this silently destroys it on every save.
   - **Recommendation:** Either preserve the existing `sort_order` on update (read it first, or accept it in `SaveScenario`), or remove `sort_order` from the schema and sort by `name` only to be explicit.

5. **[Minimalist] `_previous_scenario_name` parameter is a dead artifact**
   - **File:** `app/src-tauri/src/commands/eval_workbench/mod.rs:330`
   - **Issue:** The `save_scenario` command still accepts `previous_scenario_name: Option<String>` but prefixes it with `_` (unused). This was needed for the file-based rename logic (detecting renames, deleting old files). With SQLite, rename is just an UPDATE on `name`.
   - **Recommendation:** Remove the parameter from the command signature and all callers in a follow-up cleanup. Leaving it invites confusion about whether rename semantics still apply.

6. **[Skeptic] `parse_generated_scenario_response` silently falls back to existing data**
   - **File:** `app/src-tauri/src/commands/eval_workbench/mod.rs:141-154`
   - **Issue:** If the LLM response is missing `prompt` or `expectations`, the function falls back to `existing_scenario.prompt` and `existing_scenario.assertions`. This means a malformed or partial LLM response results in a "successful" generation that silently preserves stale data. The user sees no error but gets no new content.
   - **Recommendation:** Consider failing loudly when the LLM response is missing required fields, or at minimum log a warning so the issue is observable.

7. **[Skeptic] `delete_scenario` in DB layer doesn't verify the scenario exists before deleting**
   - **File:** `app/src-tauri/src/db/eval_workbench.rs:218-232`
   - **Issue:** The function executes `DELETE FROM scenarios WHERE ...` and returns `Ok(())` regardless of whether any rows were affected. A delete of a non-existent scenario returns success silently.
   - **Recommendation:** Check `rows_affected` from the execute call and return an error if 0 rows were deleted, matching the semantics of the old file-based delete which would error if the file didn't exist.

### Low

8. **[Minimalist] Duplicate `scenario_to_dto` bodies for `ScenarioDto` and `ScenarioSummaryDto`**
   - **File:** `app/src-tauri/src/commands/eval_workbench/mod.rs:50-70`
   - **Issue:** Both functions produce identical field mappings (id, plugin_slug, skill_name, name, tags, prompt). `ScenarioSummaryDto` includes `prompt` even though the summary type in the old code didn't. This is not wrong but adds redundancy.
   - **Recommendation:** If `ScenarioSummaryDto` doesn't need `prompt`, remove it from the DTO and the mapping.

9. **[Skeptic] `package-lock.json` changes in the duplicate-button cleanup commit**
   - **File:** `app/package-lock.json` (commit `78a1602d`)
   - **Issue:** The commit that removes the duplicate "New scenario" button also changes `dev: true` → removed for `@types/react`, `@types/react-dom`, and `csstype`. These changes are unrelated to the button removal and suggest a package manager resolution difference.
   - **Recommendation:** Verify these lockfile changes are intentional and not a side effect of running `npm install` with a different npm version.

10. **[Minimalist] Prompt template file retains old name**
    - **File:** `agent-sources/prompts/eval-workbench-suggest-scenario.txt` (referenced at `mod.rs:14-17`)
    - **Issue:** The constant was renamed from `SUGGEST_SCENARIO_PROMPT_TEMPLATE` to `GENERATE_SCENARIO_PROMPT_TEMPLATE`, but the source file is still named `eval-workbench-suggest-scenario.txt`. This is cosmetic but creates a naming mismatch.
    - **Recommendation:** Rename the file to `eval-workbench-generate-scenario.txt` in a follow-up, or document why the old name is retained.

## What Went Well

1. **Race condition fix is correct.** The commit `cda68c60` explicitly sets up event listeners *before* dispatching `send_openhands_message`, which is the right ordering. The previous version had a window where events could arrive before listeners were registered.

2. **Clean deletion of eval-run execution.** The removal of `eval-run.ts`, `eval-running-state.ts`, `RunStatusFooter`, Escape-key cancellation, and tab-switch guards is thorough. No orphaned imports or dead code paths remain in the reviewed files.

3. **DB layer is well-tested.** The `eval_workbench.rs` tests cover save/read, list filtering, update with assertion replacement, delete with cascade, and mode validation. The test `deletes_scenario_cascading_assertions` explicitly verifies the cascade behavior.

4. **Generation command fails loudly without conversation.** The error message `"No active OpenHands conversation for skill '{}' in plugin '{}'. Select a skill conversation in Refine first."` is actionable and matches the pattern used by Workflow.

## Verdict

**REQUEST_CHANGES**

Two high-severity findings block a clean approval:

1. **`workspace_run_dir` pointing to workspace root** needs verification that it doesn't cause the generation agent to write to the workspace during a `SendExistingOnly` dispatch. This is a safety concern.
2. **Listener leak on `send_openhands_message` failure** is a real resource leak that would accumulate with repeated failures.

Both are fixable with targeted changes. The medium-severity findings (unique constraint, sort_order, dead parameter, silent fallback, silent delete) should be addressed or tracked as follow-ups.

## Next Steps

1. **Fix listener leak:** Add a `Drop` impl for `TurnListener` that calls `app.unlisten()` for both listeners, or restructure `generate_eval_scenario_assertions` so listeners are always cleaned up regardless of which step fails.

2. **Verify `workspace_run_dir` safety:** Confirm that `SendExistingOnly` dispatch does not use `workspace_skill_dir` for file operations. If it does, pass a safe isolated path (e.g., a subdirectory under `.openhands/`) instead of the workspace root.

3. **Add UNIQUE constraint:** Add `UNIQUE(plugin_slug, skill_name, name)` to the `scenarios` table in migration 48 (or a new migration) to prevent duplicate names.

4. **Fix silent delete:** Return an error from `delete_scenario` when 0 rows are affected.

5. **Remove `_previous_scenario_name`:** Clean up the dead parameter from `save_scenario` command and all frontend callers.

6. **Run validation suite:** Confirm `cargo test`, `tsc --noEmit`, `vitest run`, `npm test` (contract tests), and `markdownlint` all pass on the rebased branch.
